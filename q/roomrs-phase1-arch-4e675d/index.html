05 System overview

How it all connects.

Two parallel tracks: tickets (left-to-right) and lease events (top-down). Both end with a human action — agent reply or tenant SMS. Supabase + the admin dashboard sit alongside, capturing state for both.

AI-augmented path Standard data flow Read / write to state
Flow 01 — Triage

Every ticket. Classified. Drafted. Reviewed.

WhatsApp gets synced into Desk by the whatsapp_sync Modal service. Every new or updated ticket triggers triage, which calls Claude to classify (Leasing / Maintenance / Move-In / Other), draft a reply, and detect emergency. Results written back to Desk custom fields. The router then decides: high-confidence + non-urgent → AI sends; low-confidence OR urgent → escalates to an agent in Desk.

What goes in

Per ticket
  • Channel + raw message
  • Tenant + unit from CRM
  • Recent ticket history
  • Prompt version (tracked)

What Claude does

Sonnet 4.6 · ~2s
  • Pick category (4 options)
  • Confidence 0.0 – 1.0
  • Draft reply, tone-matched
  • Flag emergency yes/no + confidence

Routing rules

Tunable in dashboard
  • Confidence ≥ threshold AND not urgent → AI sends
  • Confidence < threshold OR urgent → escalate to agent
  • Every send (AI or human) logged in triage_runs
  • Agent override on AI-sent reply → trains classifier
Flow 02 — Off-hours

After 6pm. Emergencies still get a human.

The same triage service that classifies tickets also checks the clock. If it's outside 9–6, every ticket auto-acknowledges. High-confidence emergencies (configurable keywords + Claude's judgment) page the on-call agent via Zoho Voice. Everything else queues for morning. Target SLA: page within 60 seconds.

Configurable in dashboard

Roomrs controls
  • Business hours per day (tz aware)
  • Emergency keywords list
  • Confidence threshold (default 0.7)
  • On-call rotation (round-robin or schedule)

Pager mechanics

Zoho Voice
  • SMS to on-call agent
  • Logged to escalations table
  • Max 3 retry attempts
  • Auto-ack to tenent in parallel

Audit trail

Always
  • Triggered at · acked at · who
  • Resolved at (manual close)
  • Zoho Voice message ID stored
  • Visible in escalation log
Flow 03 — Entry

Lease event → three SMS. Hands-free.

A lease is signed or move-in is scheduled in Buildium. BuildiumAdapter normalises the event, entry_flow generates/retrieves the entry code, composes instructions, and schedules three SMS via Zoho Voice: T-24h, T-2h, on arrival. State + delivery logs in Supabase.

Trigger

Buildium
  • Lease signed
  • Move-in scheduled
  • Webhook · or 15m poll fallback

State machine

Supabase
  • entry_flows: lease · status · code
  • entry_messages: per-send log
  • Cancel + reschedule on lease change

Failure handling

Built-in
  • SMS delivery failure → alert agent
  • No reply on arrival → flag for check
  • Full audit per move-in

12 Risks & open decisions

What could go wrong.

Six known risks, each with a mitigation. Four decisions still open — none block the start.

Risk 01 · timeline

Meta WhatsApp onboarding slips past 3 weeks

The single biggest variable in the timeline.

Mitigation → Ship Desk-only triage as soft-launch in Wk 4 even if WA isn't live.
Risk 02 · integration

Buildium API limits or no webhook coverage

Some events may not be webhookable.

Mitigation → Confirm in Phase 0. Fall back to 15-min polling.
Risk 03 · model quality

Low volume = slow classifier training feedback

30 tickets/day is not a fast loop.

Mitigation → Calibrate on 500-ticket historical sample. Tune live for 2 wks.
Risk 04 · product

Emergency definition is product work, not code

Needs Roomrs' input to scope what "emergency" actually means.

Mitigation → Workshop with Roomrs in Phase 0 · Day 1.
Risk 05 · future-proofing

Buildium swap mid-project

You've mentioned a possible PMS change later.

Mitigation → PMSAdapter absorbs most of it. ~30–60 hrs for the second adapter.
Risk 06 · platform

Zoho Desk custom field limits on current plan

We need a handful of new fields (ai_confidence, suggested_response, etc.).

Mitigation → Verify in Phase 0. CRM Plus should be fine.
Open decisions — none block start

Principles

Five decisions that keep Phase 2 cheap.

Every decision is reversible. None of them lock Roomrs into our layer.

  • Augment, don't replace. Agents stay in Desk. Tickets stay in Desk. Tenants stay in CRM. We add a layer next to it — removable without data loss.
  • One vendor per role. Zoho Voice for SMS in both flows. Twilio only handles WhatsApp transport. No telephony sprawl.
  • Buildium behind a PMSAdapter. Future PMS swap = config change + second adapter, not a rebuild.
  • Pay-per-use compute. Modal scales to zero. Vercel scales by request. No idle infrastructure billed.
  • Confidence-gated autonomy. High-confidence non-urgent replies AI-send; low-confidence or urgent tickets escalate to an agent. Threshold is tunable. Every send is logged. Phase 2 widens the autonomy band using the confidence data captured here.
